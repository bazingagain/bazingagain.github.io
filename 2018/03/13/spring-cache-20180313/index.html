Spring学习之缓存的使用

发表于 2018-03-13 | 分类于 Spring | | 阅读次数 | 访问次数 次
字数统计 1,314 | 阅读时长 6

为什么要使用缓存？

通常，在项目中，有许多数据一般变更较少，但又是查询量非常大的热点数据，这个时候，如果每次查询都去数据库中查询一遍再返回给用户，是非常消耗系统资源且浪费时间的。

而每次把这些热点数据缓存起来放到内存中，再次查询时直接从内存中取出，那么就会快很多，提高了查询的效率。

在Spring中是怎样使用缓存的？

在Spring搭建项目中，Spring提供多种缓存方案供我们选择使用：

  • SimpleCacheManager
  • NoOpCacheManager
  • ConcurrentMapCacheManager
  • CompositeCacheManager
  • EhCacheCacheManager
  • RedisCacheManager (Spring Data Redis)
  • GemfireCacheManager (Spring Data Redis)

其中，ConcurrentMapCacheManager,EhCacheCacheManager,RedisCacheManager的使用较为常见。

Spring对Cache的管理最顶层是提供了CacheManager接口，由一系列的实现完成对缓存的管理，我们选择不同的方案无非是提供不同的CacheManager的实现（即提供不同的CacheManager的具体实现bean）。

Spring对缓存的使用做了非常好的统一工作，这样我们不管使用哪种缓存配置方案，只需更改CacheManager的bean的声明，而无需更改缓存使用的具体方式。

缓存使用的细节

Spring缓存的Java配置方式中，提供了@Cacheable,@CachePut,@CacheEvict和@Cacheing注解：

@Cacheable - 调用方法前，会首先从缓存中找返回值，如果没有，则调用方法，并将返回值放入缓存中

@CachePut - 首先调动方法，并将返回值放入缓存中

@CacheEvict - 从缓存中清除一个或多个条目

@Cacheable 和 @CachePut共有的属性：

  1. value : String[]，要使用的缓存名称
  2. condition : String, SpEL表达式，若值为false，则不将缓存应用到方法调用上
  3. key : String, SpEL表达式，计算自定义的缓存key
  4. unless : String, SpEL表达式， 若值为true，则返回值不会放到缓存中

@CacheEvict属性：

  1. value : String[], 要使用的缓存名称
  2. key : String, SpEL表达式，计算自定义的缓存key
  3. condition : String, SpEL表达式，若值为false，则不将缓存应用到方法调用上
  4. allEntries : boolean, 若为true，特定缓存的所有条目都被移除
  5. beforeInvocation : boolean, 若为true，则在方法调用之前移除条目，如果为false(默认)，则在方法调用之后移除条目

Spring提供的自定义缓存规则的SpEL扩展：

表达式 描述
#root.args 穿肚给缓存方法的参数，形式为数组
#root.caches 该方法执行时所对应的缓存， 形式为数组
#root.target 目标对象
#root.targetClass 目标对象的类
#root.method 缓存方法
#root.methodName 缓存方法的名字
#result 方法调用的返回值(不能用在@Cacheable上）
#Argument 任意方法参数名或参数索引

unless 与condition的不同点：

unless只能阻止将对象放进缓存，但当方法调用时，依然会去缓存中查找，找到就返回要的值。
而condition计算的结果为false的话，那么这个方法被调用的过程中，缓存是被禁用的，即既不去缓存中查找，也不讲返回值放入缓存。
